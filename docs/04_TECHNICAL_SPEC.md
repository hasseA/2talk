# Technical Specification

## 1. Purpose

This document describes the technical architecture of the 2talk MVP.

The implementation should prioritize simplicity, maintainability, and reliability over unnecessary complexity.

The objective is to build one complete working product rather than a large platform.

---

# 2. System Architecture

The application consists of four major components:

* Frontend
* Backend API
* Database
* OpenAI API

```text
Browser
    │
    ▼
Frontend
    │
    ▼
Backend API
    │
    ├──────── Database
    │
    └──────── OpenAI API
```

The frontend never communicates directly with OpenAI.

All AI communication is performed by the backend.

---

# 3. Frontend Responsibilities

The frontend is responsible for:

* displaying the user interface
* creating conversations
* joining conversations
* displaying mediated messages
* collecting user input
* displaying private AI guidance
* updating the conversation in real time

The frontend should remain lightweight.

Business logic belongs in the backend.

---

# 4. Backend Responsibilities

The backend is responsible for:

* conversation creation
* invitation management
* participant management
* conversation security
* AI communication
* mediation processing
* language translation
* conversation storage
* summary generation

The backend is the only component allowed to access the OpenAI API key.

---

# 5. Database

Minimum database tables:

## conversations

Stores:

* conversation id
* title
* invitation token
* status
* created date

---

## participants

Stores:

* participant id
* conversation id
* display name
* preferred language
* joined date

---

## messages

Stores:

* message id
* conversation id
* sender id
* original message
* mediated message
* original language
* delivered language
* created date

The original message is stored for audit purposes but is never displayed to the other participant.

---

## summaries

Stores:

* conversation id
* AI summary
* created date

---

# 6. AI Processing

Whenever a participant sends a message:

1. Save the original message.
2. Send the conversation context to OpenAI.
3. Generate the mediated message.
4. Translate if necessary.
5. Store the mediated version.
6. Deliver the mediated message to the recipient.
7. Store any private guidance generated by the AI.

Every message follows this pipeline.

---

# 7. OpenAI Integration

The backend should use a single OpenAI request for each message.

The AI should receive:

* conversation history
* sender language
* recipient language
* system instructions
* current message

The AI should return structured JSON rather than plain text.

Example fields:

* mediated_message
* sender_guidance
* recipient_guidance
* detected_language
* emotion
* communication_goal
* requires_pause
* conversation_summary (when requested)

Using structured output makes the backend predictable and easier to maintain.

---

# 8. Security

The MVP should include basic security.

Requirements:

* secure invitation tokens
* HTTPS
* server-side API key storage
* input validation
* output validation
* protection against unauthorized room access

The frontend must never expose the OpenAI API key.

---

# 9. Conversation History

Every AI request should include enough recent conversation history to preserve context.

The backend may limit the history window if necessary for performance.

Conversation history should always remain in chronological order.

---

# 10. Real-Time Updates

Messages should appear without requiring users to refresh the page.

The implementation may use:

* WebSockets
* Server-Sent Events (SSE)
* polling

The chosen approach should favour simplicity and reliability for the MVP.

---

# 11. Error Handling

If the AI service is temporarily unavailable:

* the original message must remain stored,
* the sender should receive a clear notification,
* the message should not be delivered until mediation succeeds.

The system should never bypass mediation by sending the raw message directly.

---

# 12. Future Expansion

The architecture should allow future support for:

* additional AI providers
* voice conversations
* video conversations
* user accounts
* mobile applications
* group conversations
* advanced moderation
* enterprise deployments

These features are intentionally excluded from the MVP but should not require a complete redesign later.
